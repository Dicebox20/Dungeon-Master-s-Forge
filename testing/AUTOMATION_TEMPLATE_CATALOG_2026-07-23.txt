Dungeon Master's Forge Automation Template Catalog

These are automation templates, not item-family templates. An item-family template answers “what Foundry document is this?” An automation template answers “what behavior is this, when does it run, who does it affect, what does it spend, and how do we prove it worked?”

PRODUCTION TEMPLATES

1. workflow-condition-rider
Category: workflow-rider
Capability IDs: C04, C05
Recipe: conditionOnHit
Trigger: attack-hit
Target model: hit targets, optionally filtered by creature type
Required fields: condition, save, duration, targetFilter
Route: Midi-QOL + Item Macro when advertised; core attack workflow with review otherwise
Proof: save is an object; condition reaches only eligible targets; duration is preserved and expires

2. self-token-light-toggle
Category: stateful-self-effect
Capability IDs: C03, C10, C14
Recipe: selfTargetLight
Trigger: utility activation
Target model: wielder's actor token
Required fields: bright light, dim light, duration, toggle state
Route: Item Macro when advertised; portable light metadata with review otherwise
Proof: the actor token is the source; the toggle is reversible; no unrelated target dialog appears

3. shared-charge-activity-set
Category: resource-and-activity
Capability IDs: C01, C02, C11
Recipe: multiActivityResource
Trigger: named activity activation
Target model: defined by each activity
Required fields: named activities, charge cost, shared uses, recovery
Route: native DND5e
Proof: every named power has its own activity; the intended charge pool is spent once; recovery is stored once

4. attunement-effect-transfer
Category: passive-transfer
Capability IDs: C06, C08, C13
Recipe: daeTransferEffect
Trigger: effect application
Target model: equipped item or actor
Required fields: effect changes, duration, transfer target, attunement gate
Route: DAE when advertised; native effect data with review otherwise
Proof: approved system paths are used; attunement controls transfer; removal is idempotent

QUESTIONNAIRE RULE

For every prompt containing more than one mechanic, record one block per capability. Include the automation template ID when a production template is selected, the route and required modules, the target/source, resource owner, lifecycle, structural result, live result, review-note result, and final status.
